# 2026 暑期路線：資料結構凍結與 Schema 檢查器

> 2026-07-11 討論定案的方向紀錄。背景：以四份真實問卷（去識別化 demo）
> 對照程式碼完成資料結構文件（見 `dataformat.md`），並就「資料結構還能不能動」
> 達成結論。

## 決策：線上格式凍結

對照表單前 8 列 meta＋第 9 列名冊的列結構、type/format 單字母矩陣、
content 各套 mini-grammar、紀錄表 A–E 欄佈局——同時被**問卷作者的使用習慣**、
Code.js、匯出工具＋歷史紀錄表三方依賴，改動＝遷移所有現存問卷＋重訓作者，
收益只有「格式變漂亮」，不做。

實務上問卷都是一次性的（一個活動一組 refer+record），「上線後改欄位導致紀錄
錯位」的風險被使用模式天然繞開，凍結的代價可承受。

投資方向改為三層：

1. **凍結不動**：上述線上格式。
2. **邊界防護**（本計畫主體）：把隱式格式規則變成可執行的檢查。
3. **向後相容擴充**（備忘，有需求再做）：問卷列表 P 欄之後加欄、新增 format
   字母、紀錄表**尾端**加 schema 指紋欄（寫入當下的欄位 ID 清單，讓匯出工具
   偵測錯位）——都不影響舊表。

## 主項目：對照表單 Schema 檢查器

**落點（已定案）**：「問卷列表」試算表的 **container-bound GAS**（跟輸出主控台／
執行紀錄同一個腳本專案），不放 web app 也不放 tools/。理由：

- 檢查時機是「問卷作者編完對照表單、上線前」，作者的工作介面就是問卷列表那張表；
  container-bound 可以做 onOpen 自訂選單，點一下就檢查。
- web app 端做 runtime 檢查對填表人太晚（表已上線）也拖慢載入。
- tools/ 的本機腳本問卷作者不會跑。

**檢查規則草案**（每條都有真實案例背書，細節見 `dataformat.md`）：

| 規則 | 案例來源 |
|------|----------|
| 非法 type×format 組合（合法矩陣見 dataformat.md） | 目前只存在 Code.js if-chain，填錯無回饋 |
| content 解析失敗：`L` 非三段、`X` 非四段、`F` 上傳四段式缺段 | — |
| `P` 郵遞區號第二段引用的欄位 ID 不存在 | `3;F01,F02,F03` 引用其他欄 |
| C-S 計算欄引用的欄位 ID 不存在 | `N01:80;N02:180;N03:180` |
| 欄位 ID 重複 | 真實填入表出現過 `S01` 兩次 |
| 名冊殘留 Excel 錯誤值（`#N/A` 等） | 選班群表 C25 欄公式殘留 |
| meta 列缺漏（前 8 列有空該有值的格） | — |
| 主鍵欄（type P）缺失或多於一個 | 登入依賴唯一主鍵 |
| 名冊主鍵值重複 | 同人兩列會登入到第一列 |
| 紀錄表 F 欄起的標題列與對照表單欄位對不上（錯位預警） | 真實填入表與問卷欄位曾不同步 |
| 問卷列表布林欄填了「是／否」以外的值 | 各欄真值判定不一致（`≠否` vs `＝是`），填 Y/TRUE 兩樣情 |
| 截止時間戳 parseInt 出來是 NaN 或早於現在 | 手填毫秒 timestamp 易錯 |

**細節定案＋實作（2026-07-11）**：已實作於 `tools/export.js`（即 container-bound
專案的原始碼，手動貼上，不隨 clasp 部署）。觸發＝onOpen「問卷管理」選單
（GAS 沒有右鍵選單 API，以「點選列＋功能表」代替 context menu）；結果呈現＝
ui.alert（錯誤／警告兩級，超過 45 行截斷）。同次實作：「新增問卷」（建立前強制
過檢查——錯誤擋下、警告需確認放行；自動建立填入表＋列表新列，顯示／開放進入
預設「否」）、「輸出問卷」改為點選列＋問答驅動（退役「輸出主控台」分頁與
`listSheet` ScriptProperty，改用 getActiveSpreadsheet）。紀錄表錯位檢查靠填入表
第 1 列的欄位 ID 標題比對——新增問卷會自動寫入這列，等同輕量版 schema 指紋。
**待人工驗收**：貼上 container-bound 專案實測三個選單項（見 tools/README.md）。

## 問卷工具第二波：資料夾屬性＋鷹架建卷＋欄位輔助精靈（2026-07-11 設計定案）

第一波（選單四功能）之後的擴充，全部在 container-bound GAS（tools/export.js），
零 web app 改動。核心哲學定案：**不做網頁式整卷編輯器——Google Sheet 就是
編輯器本體**（排序/插刪/複製它都免費），精靈只補「記格式參數」這個最弱環節。

### 1. ScriptProperties 擴充（只加需要的，嚴禁鏡射 web app 屬性）

- `referStorage`：結構表（對照表單）資料夾 ID
- `recordStorage`：寫入表（填入表）資料夾 ID
- 既有 `exportStorage` 不動。**`jwtSecret` 等 web app 屬性絕不複製過來**
  （工具用不到，多一份多一個洩漏點）；`listSheet` 已退役。
- **真正的紅利是權限一勞永逸**：兩個資料夾各共享一次給 web app 執行帳號
  （結構表資料夾檢視權、寫入表資料夾編輯權），之後建立的表自動繼承，
  「新增問卷後手動共享」那步永久消失。

### 2. 新增真的問卷（鷹架建卷）

現有「新增問卷」（貼既有對照表單 ID）保留；新增從零開始的入口：

- 在 `referStorage` 建結構表骨架：8 列 meta 預填一組示範欄位
  （分組／主鍵／常見題型各一，照 dataformat.md），作者在 Sheet 裡手動改
- 寫入卷建在 `recordStorage`（標題兩列同現行）並 **`protect()` 保護：
  編輯者只留 web app 執行帳號＋擁有者**——「不准改寫入卷」從紀律變機制
- 掛上列表新列（顯示／開放進入預設「否」）

### 3. 欄位輔助精靈（新選單項）

HtmlService 對話框，勾選生成單一欄位的 8 格 meta（暱稱：欄位 EXIF）：

- **題型用人話選**（文字／長文字／身分證／手機／Email／郵遞區號／滑桿／下拉／
  多選／檔案上傳／說明區塊／計算欄／檔案檢視／主鍵／認證／輸出／分組），
  精靈負責翻譯成 type×format＋content mini-grammar，出題者不碰語法
- 參數表單依題型動態出現（選項清單、滑桿範圍、檔案限制、正則樣板＋自訂）
- 旗標勾選：必填 `M`／可留白 `N`／可宣告無資料 `D`（Phase 15）／群組名＋`:U`
  ——「手機可填無」＝format M＋勾 D，**不再生成「內建正則＋`|無`」**，
  regex 只剩真客製（含字、日期等樣板與自訂）才出現
- **產出前過檢查器同一套驗證**（`checkContent_`／`LEGAL_FORMATS_` 同專案直接呼叫），
  精靈生成的欄位保證檢查器全綠
- 輸出：8 行直排文字＋複製按鈕（多行貼上在 Sheets 自動直排填 8 格，一欄一貼）；
  若當下有點選問卷列，另提供「直接寫入該卷對照表單的下一個空欄」

### 4. 修改問卷（選單項）

- 開啟點選列的對照表單連結；改完自動跑格式檢查
- 寫入卷已有紀錄時明確警告：文案類修改（說明／選項／正則）安全，
  結構類（增刪挪欄）會錯位——檢查器的標題比對是防線
- 明確不做：網頁式整卷編輯器（理由見上）；欄位「讀回精靈編輯」先不做，
  改欄位＝精靈重生成一欄貼上覆蓋

## 次項目（低成本，不動格式）

1. **`compareSheets` off-by-one 統一**：`splice(0,7)` 把第 8 列（nullable 列）
   留在名冊資料裡，與 `getHeaders`／`buildSelections` 的「第 9 列起」不一致，
   目前靠第 8 列分組欄恰為空字串沒炸。統一成第 9 列起。
2. **主登入節流**：主鍵／認證欄是低熵個資（學號、身分證），登入比對目前無
   嘗試次數限制，可枚舉。比照邀請 OTP 的「連錯 N 次作廢／冷卻」機制。

## 明確不做

- 改 meta 列結構、換 content 語法、改紀錄表 A–E 欄佈局（凍結決策）。
- 移除 group 的 `:U`（群組值不重複）：程式有實作（`columnPrep`／`validateColumn`）
  但實務從未使用——保留不動，已在 dataformat.md 標註。
